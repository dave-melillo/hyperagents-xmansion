## 1. Problem Statement

**What problem are we solving?**
X-Mansion agents currently learn in isolation. When Beast discovers that structured templates improve PRD pass rates, that learning stays in Beast's memory files. Nightcrawler, Wolverine, and other agents don't benefit. Skills degrade over time without feedback loops, and there's no systematic way to identify underperforming skills or propagate successful patterns across the team.

**Why now?**
Meta AI's Hyperagents paper (March 2026) provides a proven framework for self-improving agent systems. The core patterns (metrics collection, cross-agent learnings, meta-agent reviews) can be adapted to our existing OpenClaw architecture at low cost (~$20/month vs $500/experiment for full DGM-H). This is foundational infrastructure that will compound benefits across all future work.

**Who is affected?**
- **Primary:** All X-Men agents (Beast, Wolverine, Cyclops, Nightcrawler, Gambit, Forge, Storm, Professor X)
- **Secondary:** Dave (visibility into system health, reduced manual intervention)
- **Context:** Agents execute skills during task workflows via OpenClaw. Improvements happen automatically via heartbeats and cron jobs.

---

## 2. Solution Overview

**One-liner:** Add metrics tracking, cross-agent learning capture, and automated meta-reviews to enable systematic self-improvement across the X-Mansion team.

**Approach:**
We adapt three core patterns from Hyperagents:

1. **Metrics Infrastructure:** Each skill gets a `metrics.json` file tracking invocations, success/failure rates, common failure patterns, and learnings. A `log-skill-outcome.sh` script makes logging easy.

2. **Cross-Agent Learnings:** A shared `~/moltbot/skills/_meta/learnings.md` file captures patterns that transfer across agents. Agents extract learnings after completing tasks and append them here. Any agent can read this before starting work.

3. **Meta-Agent Reviews:** Forge runs weekly automated reviews, scanning skill metrics for underperformers (<75% pass rate), analyzing failure patterns, and proposing SKILL.md improvements. Changes require approval via #blackbird thumbs-up before merging.

**Key Design Principles:**
- **Cheap over expensive:** Adapt patterns, don't port the full $500/experiment system
- **Git-tracked everything:** All changes versioned, rollback always possible
- **Human-in-the-loop for structural changes:** Approval required before modifying skills
- **Compound value:** Investments in infrastructure benefit all future work

## 3. User Stories

### Primary Flows

1. As an **agent completing a skill-based task**, when I finish execution, I want to log the outcome (success/partial/failure) so that metrics accumulate for analysis.

2. As an **agent discovering a useful pattern**, when I find something that worked well, I want to append it to shared learnings so that other agents benefit.

3. As **Forge during weekly review**, when I scan skill metrics, I want to identify underperformers and propose improvements so that skills continuously improve.

4. As **Dave checking system health**, when I want to understand skill performance, I want to see metrics dashboards so that I know where to focus attention.

### Secondary Flows

5. As an **agent starting a task**, when I begin work using a skill, I want to check learnings.md for relevant patterns so that I avoid known pitfalls.

6. As an **agent needing to rollback**, when a skill change causes problems, I want to restore a previous version so that nothing breaks permanently.

### Edge Cases

7. As an **agent with conflicting learnings**, when two patterns contradict, I want clear precedence rules so that I know which to follow.

8. As an **agent with a failing skill**, when metrics show <50% pass rate, I want the skill flagged for urgent review so that it gets fixed quickly.

## 6. Algorithm & Logic Specifications

### 6.1 Skill Health Scoring

**Purpose:** Quickly identify which skills need attention.
**Priority:** High (drives Forge's review focus)

**Inputs:**
- invocations (total count)
- outcomes.success, outcomes.partial, outcomes.failure
- common_failures (patterns)

**Logic:**
```
pass_rate = (success + 0.5 * partial) / invocations * 100

IF invocations < 5:
  health_status = "insufficient_data"
ELIF pass_rate >= 75:
  health_status = "healthy"
ELIF pass_rate >= 50:
  health_status = "needs_review"
ELSE:
  health_status = "critical"
```

**Output:** health_status enum, pass_rate percentage

**Display Rules:**
- healthy: Green badge ✅
- needs_review: Yellow badge ⚠️
- critical: Red badge 🔴
- insufficient_data: Gray badge ❔

---

### 6.2 Learning Extraction

**Purpose:** Capture transferable patterns from task completions.
**Priority:** Medium (runs after every significant task)

**Inputs:**
- Task outcome
- Agent reflection on what worked/failed

**Logic:**
```
Agent asks itself after task completion:
1. Did I discover a technique that made this easier?
2. Did I encounter a failure pattern others should avoid?
3. Did I find a tool combination that worked particularly well?

IF any answer is yes:
  Extract learning with:
    - date: today
    - agent: self
    - skill_context: skill used
    - category: classify as prompt_engineering|tool_usage|failure_patterns|process
    - learning: 1-2 sentence summary
    - evidence: what happened
  
  Append to ~/moltbot/skills/_meta/learnings.md
```

**Output:** New learning entry in shared file

---

### 6.3 Forge Meta-Review

**Purpose:** Systematically identify and improve underperforming skills.
**Priority:** High (runs weekly via cron)

**Inputs:**
- All skill metrics.json files
- Shared learnings.md
- Skill SKILL.md files

**Logic:**
```
1. SCAN all ~/moltbot/skills/*/metrics.json
2. FILTER skills where health_status in ["needs_review", "critical"]
3. For each underperforming skill:
   a. READ SKILL.md
   b. ANALYZE common_failures patterns
   c. CHECK learnings.md for applicable patterns
   d. PROPOSE specific improvements (patch format)
4. GENERATE report with:
   - Skills reviewed
   - Proposed changes (diff format)
   - Expected impact
   - Request for approval
5. POST to #blackbird
6. AWAIT thumbs-up before applying any changes
```

**Output:** Review report with actionable proposals

---

### 6.4 Skill Versioning

**Purpose:** Enable rollback if changes cause problems.
**Priority:** Medium

**Inputs:**
- Current SKILL.md
- Trigger (manual or post-improvement)

**Logic:**
```
1. GENERATE version number:
   - Major: breaking changes to skill interface
   - Minor: new capabilities or significant improvements
   - Patch: bug fixes, clarifications
2. COPY SKILL.md to versions/vX.Y.Z.md
3. UPDATE metrics.json with:
   - version_checkpointed: vX.Y.Z
   - pass_rate_at_checkpoint: current_pass_rate
4. COMMIT with message: "checkpoint: skill-name vX.Y.Z"
```

**Output:** Versioned snapshot in skill's versions/ directory

### Stack

| Layer | Technology | Rationale |
|---|---|---|

| Scripts | Bash + jq | Consistent with existing skills, no new deps |

```
└─────────────────────────────────────────────────────────────┘

│                    #blackbird Channel                        │
│  Review report → Human approval → Apply changes              │
└─────────────────────────────────────────────────────────────┘
```

### Key Technical Decisions

1. **JSON files over database:** Simpler, git-trackable, no infrastructure. Risk: concurrent writes could conflict. Mitigation: Atomic file operations via temp file + mv.

2. **Bash scripts over Python:** Consistent with existing skill scripts, no additional dependencies. Risk: Complex logic harder in bash. Mitigation: Keep scripts focused, use jq for JSON.

3. **Human approval for changes:** Prevents automated damage. Risk: Bottleneck on Dave. Mitigation: Batch weekly reviews, clear approval UX (reactions).

## Appendix A: Research Findings

### Hyperagent Patterns Applicable to X-Mansion

1. **Archive of Stepping Stones → Skill Versioning**
   - Hyperagents maintain a population of successful agent variants
   - We adapt this as git-versioned skill snapshots
   - Enables rollback and A/B comparison

2. **Performance Tracker → Skill Metrics**
   - Hyperagents autonomously create tracking classes
   - We implement as metrics.json per skill
   - Captures success rates, failure patterns, learnings

3. **Meta Agent → Forge**
   - Hyperagents have a meta agent that modifies task agents
   - We designate Forge as the meta-agent role
   - Weekly reviews, proposes improvements, requires approval

4. **Self-Modification via Bash Tool**
   - Hyperagents edit code via view/create/str_replace
   - We already have this via edit tool
   - Add protocol for when to self-modify vs request review

### Cost Analysis

| Approach | Cost per Cycle | Tokens | Suitability |
|----------|---------------|--------|-------------|
| Full DGM-H | ~$500 | ~88M | Research only |
| Our Adaptation | ~$5-20/month | ~500K | Production suitable |

### Cross-Domain Transfer Evidence

From the paper: "Meta-level improvements transfer across domains and accumulate across runs."

Example: Strategies learned for robotics reward design improved math grading by 15%.

We can achieve similar transfer via shared learnings.md.
